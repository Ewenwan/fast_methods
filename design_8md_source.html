1 # Library design
2 We provide here a small overview of the software design choices.
3 
4 ## Criterions
5 - Code easy to use for the user.
6 - Avoid long and nested namespaces and variable names.
7 - Create a hierarchy that allows to reuse the code.
8 - Do not rely on weird C++ tricks.
9 - Do not rely on too many dependencies.
10 - Code flexibility and high performance.
11 
12 ## Design
13 - To reach high performance, `std::array` are mainly used.
14 - To avoid many dependencies, C++11 was chosen.
15 - To make the code easier to use, we only rely on Boost libraries and CImg (for visualization)
16 - To reuse code as much as possible, solvers are grouped depending on how they work. FMM-based solvers have many common procedures. The same as FM2-based solvers.
17 - Also, we tried to use the `information expert` pattern.
18 - In order to reach the flexibility and high performance, we chose to use a combination of static (templates) and dynamic polymorphism. That is, all solvers inherit from the Solver class. However, Solver itself is templated depending the grid type.
19 
20 This way, the compiler can carry out tons of optimizations by knowing in advance the number of dimensions on the grid. Actually, presumably the performance is the same as if the operations were hardcoded for any number of dimensions.
21 
22 FMM-based solvers, following the `policies` design patter, have other parameter templates that change the behaviour. Concretely, the heap types.
23 
24 The dynamic polymorphism allows to use all solvers under a common interface, as the examples included.

28 ## Folder structure
29 + __cmake__: cmake helper files and modules.
30 + __data__: files and maps to test the code.
31  + __alpha__: contains code the be refactorized and included in the library in the future.
32 + __doc__: doxygen documentation.
33 + __examples__: examples to understand how to use the library.
34 + __include/fast_methods__: header files with most of the code, as the library is mostly templated.
35  + __benchmark__: code for the benchmark tool.
36  + __console__: the console class, useful for printing and parsing arguments.
37  + __datastructures__: data types of the Fast Marching Method and other algorithms such as heaps or queues.
38  + __fm__: Fast Methods Algorithms.
39  + __fm2__: Fast Marching Square (FM2) algorithms.
40  + __gradientdescent__: gradient descent implementation to compute shortests paths from the Fast Methods outputs.
41  + __io__: input/output and visualization helper classes.
42  + __ndgridmap__: main container and types of the Fast Methods.
43  + __thirdparty__: others' software used.
44  + __utils__: utility functions used in the library.
45 + __scripts__: matlab scripts to parse outputs and automatic benchmarking bash script.
46 + __src__: source code for non-templated classes.
